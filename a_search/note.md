# 学习笔记：寻路算法启发式策略性能深度分析

## 1. 问题描述 (Problem Description)

在复杂的二维栅格地图（存在 U 型墙和条形障碍）中，我们需要寻找从**绿色起点**到**红色终点**的最短路径。本次实验的核心是对比**启发式信息 (Heuristics)** 如何影响搜索效率。

### 核心对比项：
* **A* (Manhattan)**：基于 $L_1$ 范数，计算水平和垂直步数之和。
* **A* (Euclidean)**：基于 $L_2$ 范数，计算两点间的直线距离。
* **Dijkstra**：无启发式信息（相当于 $h(n)=0$），仅依赖已走代价进行盲目扩张。

---

## 2. 算法原理与实现分析 (Analysis & Logic)

### 估价函数公式
所有的搜索逻辑都遵循：$$f(n) = g(n) + h(n)$$
* **$g(n)$**: 从起点到当前格子的实际步数。
* **$h(n)$**: 预测到终点的距离。

### 启发式函数实现 (Python)
```python
# 曼哈顿距离：与 4-neighbor 移动模式完全匹配
def manhattan(p1, p2):
    return abs(p1[0] - p2[0]) + abs(p1[1] - p2[1])

# 欧几里得距离：直线距离，总是小于或等于实际网格步数
def euclidean(p1, p2):
    return math.sqrt((p1[0] - p2[0])**2 + (p1[1] - p2[1])**2)

# Dijkstra：不提供任何预测，h 恒为 0

    return 0
```

## 4. 深度分析 (In-depth Insights)

### 为什么 Manhattan 表现最好？
由于本程序限制为 **四邻域移动**（上下左右），曼哈顿距离能最精确地反映“如果没有障碍，我最少还要走几步”。
* 它提供的 $h(n)$ 最接近真实值，使得算法能够显著过滤掉背离目标的节点。

### 为什么 Euclidean 效率中等？
欧几里得距离计算的是直线距离。在只能走直角弯的网格中，直线距离总是**保守估计（Underestimating）**了剩余路程。
* 这种保守导致算法在面临障碍物时，容易在目标周围进行不必要的“扇形”尝试，从而增加了 84% 的节点探索量。

### Dijkstra 的代价
Dijkstra 算法本质上是**广度优先搜索 (BFS)** 的权重版本。
* 从图中可见，它探索了 **2363** 个节点，几乎覆盖了地图中所有可达区域。
* **结论**：在明确知道目标位置的情况下，不使用启发式信息会导致巨大的性能浪费。

## 5. 总结 (Conclusion)

1.  **路径一致性**：三种方法最终找到的红色路径长度完全一致，证明了 A* 算法在 $h(n)$ 可接受（Admissible）时，依然能保证最优解。
2.  **性能梯度**：在栅格寻路任务中，性能排序为 **Manhattan > Euclidean > Dijkstra**。
3.  **实践建议**：
    * **4 连通网格**：首选 Manhattan。
    * **8 连通/自由移动**：首选 Euclidean。
    * **目标位置未知**：使用 Dijkstra。
